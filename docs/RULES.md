# Screenshot Sorter - 開発ルール

最終更新: 2026-03-17

---

## 1. 絶対ルール

1. **コード即書き禁止**: コードを書く前に必ず有識者議論（セクション2）を実施する
2. **コード保全**: 既存の動作するコードを許可なくリファクタリング・削除・名前変更しない
3. **バイアス禁止**: 技術選定・設計判断に個人的好みを持ち込まない
4. **品質妥協禁止**: 動くが汚いコードは許可しない
5. **技術的異議の明示**: 問題がある場合は根拠付きで反論する
6. **エビデンスベース**: 客観的指標（テスト結果・行数・CI）に基づいて判断
7. **スコープ厳守**: フェーズ外の要求は BACKLOG.md に追記し、現タスクに集中する
8. **ハルシネーション禁止**: ファイルパス・関数名・設定値を参照する際は実在確認してから使う

---

## 2. 有識者議論プロセス

### 2.1 基本原則

**すべての実装作業の前に、有識者を召喚し多角的に議論してから進めること。**

### 2.2 有識者の召喚

| ペルソナ | 視点 | 召喚する場面 |
|----------|------|-------------|
| モバイルアーキテクト | 全体設計・OS間差異・API設計 | 新機能追加・構成変更 |
| セキュリティ専門家 | 権限管理・ストレージアクセス・プライバシー | 外部連携・ユーザーデータ操作 |
| QAエンジニア | テスト容易性・エッジケース・回帰リスク | テスト設計・リファクタリング |
| UXデザイナー | ユーザー体験・操作フロー・タップ数 | UI変更・ワークフロー設計 |
| ビジネスアナリスト | ROI・市場ニーズ・優先順位 | 機能要否の判断・方向性決定 |

### 2.3 スコアリング基準

| # | 評価項目 | 配点 | 観点 |
|---|----------|------|------|
| 1 | 技術的妥当性 | /10 | 選定技術は適切か |
| 2 | 保守性・可読性 | /10 | 他者が理解・修正できるか |
| 3 | スコープ適合性 | /5 | 現フェーズの目的に合致しているか |
| 4 | テスト容易性 | /5 | テストが書きやすいか |
| 5 | リスク・副作用 | /5 | 障害リスクは許容範囲か |
| 6 | 費用対効果 | /5 | 実装コストに見合う価値があるか |
| | **合計** | **/40** | **32点以上で着手許可** |

## 3. セッション開始プロトコル

毎回の会話開始時に、以下をリポジトリから直接読み込む:

1. docs/RULES.md
2. docs/DEVELOPMENT_PLAN.md
3. docs/DECISIONS.md

全ファイルを raw.githubusercontent.com 経由で読みに行くこと。

---

## 4. 変更前の影響範囲宣言（Blast Radius）

コード変更の前に宣言:
1. 変更対象ファイル
2. 影響を受けるファイル
3. テストへの影響
4. ロールバック方法

5ファイル以上に影響する変更は、必ず人間の承認を得てから実行する。

---

## 5. ハルシネーションガード

- **ファイルパス**: 実際にリポジトリ内に存在するか確認
- **関数名・クラス名**: grep で実際のソースを確認
- **パッケージ名**: pubspec.yaml に記載があるか確認
- **API名**: 公式ドキュメントのURLで実在を確認

## Markdown生成時のコードブロック規則（追加: 2026-03-17）
- ヒアドキュメント（cat << EOF）でMarkdownファイルを生成する際、ファイル内容にバッククォート3連（コードブロック）を含めないこと
- コードブロックが必要な場合は4スペースインデントで代替するか、teeコマンドを使用すること
- やむを得ずコードブロックを含める場合は、python等のスクリプト経由でファイルを生成すること
